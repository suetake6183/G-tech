# G-TECH ドライブ

末武修平の情報管理システム

## ディレクトリ構造

```
/Volumes/G-TECH/
├── _inbox/                 未分類データの一時保管場所
├── 01_ナレッジベース/      仕事を通じて得た学びを抽象化
├── 02_クライアント情報/    クライアントごとに情報を整理（Git管理）
├── 03_思考と探求/          思考を研ぎ澄ます場所
├── 04_個人管理/            事務的な個人書類
└── 99_アーカイブ/          過去のデータ保管
```

## 各ディレクトリの役割

### _inbox/
- **目的**: どこに置くべきか迷った時の一時保管場所
- **特徴**: コンテキスト付きインボックス（YAMLメタデータで分類情報を保持）
- **運用**: 迷ったらここに入れ、後で `/process` コマンドで整理
- **詳細**: [_inbox/README.md](_inbox/README.md)

### 01_ナレッジベース
- **目的**: 仕事から得た知識を抽象化し、再利用可能にする
- **特徴**: データに準じて構造化（ボトムアップアプローチ）
- **内容例**: 提案パターン、価格設定の考え方、テンプレート、ベストプラクティス

### 02_クライアント情報
- **目的**: クライアント別に全情報を一元管理
- **特徴**: Gitで管理、GitHub同期、2台のMac間で共有
- **内容例**: 提案書、契約書、会議記録、戦略メモ

### 03_思考と探求
- **目的**: 思考を研ぎ澄まし、自己成長を促進
- **テーマ**: ビジネス哲学、学習記録、自己分析
- **内容例**: 読書ノート、哲学的思考、アイデア、振り返り

### 04_個人管理
- **目的**: 個人的な事務書類の管理
- **内容例**: 証明書類、税務、家族関連、契約書

### 99_アーカイブ
- **目的**: 過去のデータを安全に保管
- **運用**: 定期的に見直し、不要なものは削除

## 設計思想

1. **役割の明確化**: 各ディレクトリは単一の明確な役割を持つ
2. **有機的成長**: 構造を事前に決めず、データに準じて形成
3. **Git活用**: クライアント情報はバージョン管理と同期を実現
4. **シンプルさ**: 複雑な構造は避け、本質的な分類のみ
5. **双方向学習**: ナレッジ・思考・実行を循環させるシステム

## インプット戦略

### データの入り口

新しいデータが発生した時、以下の判断フローで配置：

```
新しいデータ
    ↓
これは何に関連？
    ↓
┌──────────┬──────────┬──────────┬──────────┬──────────┐
│プロジェクト│クライアント│クライアント│個人思考  │わからない│
│関連      │関連（PJ明確）│（PJ不明） │         │迷った   │
└──────────┴──────────┴──────────┴──────────┴──────────┘
    ↓          ↓          ↓          ↓          ↓
02.../       02.../     02.../     03.../     _inbox/
00_INPUT/   [Client]/  _inbox/    _inbox/
            00_INPUT/
```

### コンテキスト保持の原則

**場所 + メタデータ** の組み合わせで、AIが理解できるようにする：

1. **ファイル名規則**: `YYYY-MM-DD_[種別]_[内容].[拡張子]`
2. **YAMLメタデータ**: ファイル冒頭に必ず記載
3. **テンプレート活用**: `_inbox/_TEMPLATE/` のテンプレートを使用

### 処理フロー

```
インプット → 配置 → （任意のタイミング） → /process → 整理・活用
```

詳細は [_inbox/README.md](_inbox/README.md) を参照。

## 双方向学習システム

### 循環フロー

```
[01_ナレッジベース]  ←──────┐
[03_思考と探求]      ←──┐   │
         ↓              │   │
    知識の活用         │   │
         ↓          学びの抽出
[02_クライアント提案]   │   │
         ↓              │   │
    実行・結果 ─────────┘   │
         ↓                  │
    振り返り ───────────────┘
```

### 実現する仕組み

1. **振り返り**: `/reflect` コマンドでプロジェクト完了後の振り返りをファシリテート
2. **抽象化**: `/blog` コマンドでブログ/NOTE執筆を通じた一般化をサポート
3. **整理**: `/process` コマンドでインボックスのデータを整理
4. **活用**: 標準化されたナレッジをAIが高速に検索・提案

詳細は [双方向学習システム構築_実行計画.md](双方向学習システム構築_実行計画.md) を参照。
