# リポジトリ再構築計画

**作成日**: 2025年11月8日
**目的**: クライアント情報リポジトリをフラット構造に統一し、プロジェクト中心の運用を実現する

---

## 🎯 設計方針

### 確定した原則

1. **フラット構造**: CORE/BIZ/PERSONALの分類を廃止
2. **全クライアント統一**: すべて同じプロジェクト中心構造
3. **プロジェクト単位**: 実際の作業フローに合わせた構造
4. **_共通資料**: 仕分け後に本当に共通で使うものだけ残す

### 新しい構造

```
02_クライアント情報/
├── _TEMPLATE/                    # プロジェクトテンプレート
├── _共通資料/                    # 共通リソース（仕分け後に決定）
├── _ARCHIVE/                     # 旧構造の退避先
│
├── [クライアント名]/
│   ├── CLIENT_INFO.md            # 日常業務の実務情報
│   ├── 00_CORE/                  # 戦略資料と原データ
│   │   ├── CLIENT_PROFILE.md     # 戦略分析・深層理解
│   │   ├── source/               # 原データ（LINE、音声等）
│   │   └── analysis/             # 分析資料
│   └── [プロジェクト名]/
│       ├── PROJECT.md            # プロジェクトダッシュボード
│       ├── 00_INPUT/             # 収集した情報
│       ├── 01_OUTPUT/            # 生成した書類
│       └── 02_ARCHIVE/           # 不要になったファイル
│
└── README.md                     # リポジトリ全体の説明
```

**全クライアント、この構造に統一**

## 🗺️ 移行計画

### フェーズ1: _共通資料の仕分け（最優先）

#### 目的
特定クライアント・プロジェクト固有のファイルを各プロジェクトフォルダへ移動

#### 対象ファイル例
```
_共通資料/ビジネス書類/見積書/
├── サンフィールド医院御中_見積書_251007-1.pdf
    → AOMI100クリニック/サイン・床・壁張り替え工事/01_OUTPUT/クライアント向け/見積書/

├── 吉沼様_見積書.pdf
    → 吉沼/[プロジェクト名]/01_OUTPUT/クライアント向け/見積書/

├── アストライズ御中_見積書.pdf
    → アストライズ/[プロジェクト名]/01_OUTPUT/クライアント向け/見積書/
```

#### プロジェクト名の決定方針
1. 既存のプロジェクトフォルダがあれば、そこへ
2. なければファイル内容・日付から推測してプロジェクト作成
3. 不明な場合：
   - `20XX年度案件/`
   - `一般プロジェクト/`
   - `見積・請求（20XX年）/`

#### 実行手順
1. _共通資料の全ファイルをリストアップ
2. クライアント名を特定
3. プロジェクト名を特定・作成
4. 統一構造に従って移動
5. 残ったファイルを分析

### フェーズ2: フラット化（CORE/BIZ/PERSONAL廃止）

```bash
# 移動例
```

```
_ARCHIVE/20251108_旧構造/
├── CORE_戦略的パートナー/
├── BIZ_業種別クライアント/
└── PERSONAL_個人顧客/
```

### フェーズ3: 各クライアントの構造統一

#### 黒澤工務店の例（最も複雑）

**現在の構造:**
```
黒澤工務店/
├── 01_プロジェクト/
│   ├── AI導入・DX推進/
│   ├── 業務提携（SKコーム）/
│   ├── 情報システム刷新/
│   └── ...
├── 02_コミュニケーション記録/
├── 03_ナレッジベース/
├── 04_契約・請求/
└── 戦略提案_2025年11月_...md
```

**新しい構造:**
```
黒澤工務店/
├── CLIENT_INFO.md              # 実務情報
├── 00_CORE/                    # 戦略資料と原データ
│   ├── CLIENT_PROFILE.md       # 戦略分析
│   ├── source/                 # 原データ
│   └── analysis/               # 分析資料
├── AI導入・DX推進/
│   ├── PROJECT.md
│   ├── 00_INPUT/
│   │   └── ヒアリング/
│   ├── 01_OUTPUT/
│   │   ├── クライアント向け/
│   │   └── 内部用/
│   └── 02_ARCHIVE/
├── 業務提携（SKコーム）/
│   └── ...同じ構造
└── 情報システム刷新/
    └── ...同じ構造
```

**移動方針:**
- 01_プロジェクト/ の各フォルダ → プロジェクトとして独立
- 02_コミュニケーション記録/ → 各プロジェクトの 00_INPUT/議事録/ へ
- 03_ナレッジベース/ → _共通資料/KNOWLEDGE_BASE/ または 各プロジェクトへ
- 04_契約・請求/ → 各プロジェクトの 01_OUTPUT/クライアント向け/ へ
- 戦略提案_*.md → 該当プロジェクトの 01_OUTPUT/クライアント向け/提案書/ へ

### フェーズ5: _共通資料の最終整理

フェーズ1で仕分けた後に残ったファイルを分析：

**残る可能性があるもの:**
- 契約書テンプレート（汎用的なもの）
- 見積書テンプレート（空の雛形）
- SKコームのロゴ
- プロンプト集
- マニュアル

**判断基準:**
- 複数のクライアント・プロジェクトで使うか？
  - YES → _共通資料に残す
  - NO → 該当プロジェクトへ移動、またはアーカイブ

## ⚠️ リスクと対策

### リスク1: 大規模な移動でファイルが見つからなくなる
**対策:**
- 各フェーズごとに旧構造を _ARCHIVE/ に保管
- 移動ログを作成
- 段階的に実行（一気にやらない）

### リスク2: git履歴が複雑になる
**対策:**
- 各フェーズごとにコミット
- コミットメッセージを詳細に記載

### リスク3: 既存のリンクが切れる
**対策:**
- README.mdに旧構造の対応表を記載
- 必要に応じてシンボリックリンク
