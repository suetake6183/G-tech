# CLIENT_INFO_TEMPLATE.md 検証タスク - Composer実行指示書

**作成日**: 2025-11-11
**目的**: 黒澤工務店で作成したCLIENT_INFO_TEMPLATE.mdを別のクライアントで試し、テンプレートの改善点を洗い出す

---

## 📋 タスク概要

黒澤工務店で作成・改善したCLIENT_INFO_TEMPLATE.mdを、別のクライアントに適用してテストします。テンプレートの以下の点を検証してください：

1. **汎用性**: 異なる業種・規模のクライアントにも適用可能か
2. **記入しやすさ**: 情報収集・記入に迷う箇所はないか
3. **実用性**: 日常業務で本当に使える構成か
4. **過不足**: 不要なセクション、足りないセクションはないか

---

## 🎯 実行手順

### ステップ1: テスト対象クライアントの選定

以下の基準で適切なクライアントを1つ選んでください：

**選定基準**:
- ✅ 進行中のプロジェクトがある（情報が豊富）
- ✅ 黒澤工務店と異なる特性を持つ（業種、規模、取引形態など）
- ✅ すでにある程度の資料が存在する（00_COREフォルダがある、またはファイルが散在）
- ❌ _自社情報、_TEMPLATE、ARCHIVE_完了・休眠は除外

**選定方法**:
```bash
# 各クライアントフォルダの状況を確認
ls -la "/Volumes/G-TECH/02_クライアント情報/[クライアント名]/"
```

**選定結果を記録**:
- 選んだクライアント名:
- 選定理由:

### ステップ2: 既存情報の確認と整理

選定したクライアントの現状を把握してください。

**確認項目**:
1. 既存のフォルダ構造
2. 既存のCLIENT_INFO.mdの有無（Phase 0で作成されているか）
3. 00_COREフォルダの有無と内容
4. プロジェクトフォルダの数と命名
5. 散在している資料の有無

**確認コマンド**:
```bash
cd "/Volumes/G-TECH/02_クライアント情報/[クライアント名]"
tree -L 2 -a  # または ls -la
cat CLIENT_INFO.md  # 存在する場合
ls -la 00_CORE/  # 存在する場合
```

**確認結果を記録**:
- 既存のCLIENT_INFO.md: ある / ない
- 00_COREフォルダ: ある / ない

- 主な課題:

### ステップ3: テンプレートの適用

**テンプレートを使って新しいCLIENT_INFO.mdを作成**:

1. テンプレートを読み込む
```bash
cat "/Volumes/G-TECH/02_クライアント情報/_TEMPLATE/CLIENT_INFO_TEMPLATE.md"
```

2. クライアント固有の情報を記入
   - 基本情報（会社名、業種、担当者など）
   - 事業主体（末武個人 / SKコーム / 両方）
   - 取引履歴
   - プロジェクト一覧
   - よく使う単価
   - 次回提案のアイデア

3. 実際のフォルダ構造を反映
   - 00_COREの実際の構造
   - プロジェクトフォルダのパス

4. 書き込み
```bash
# 既存のCLIENT_INFO.mdがある場合はバックアップ
if [ -f "CLIENT_INFO.md" ]; then
  cp "CLIENT_INFO.md" "CLIENT_INFO_旧版_backup.md"
fi

# 新しいCLIENT_INFO.mdを作成
# （Readツールで読み込んだテンプレートを元に、情報を埋めて書き込む）
```

### ステップ4: 検証フィードバック

## テンプレート検証結果

### テスト対象クライアント
- **クライアント名**: DOA
- **業種**: ホストクラブ経営
- **特性**: 
  - 黒澤工務店とは異なる業種（サービス業 vs 建設業）
  - 個人事業主（法人ではない）
  - 17年来の元同僚関係という特殊な関係性
  - ITリテラシーが非常に低い（1ヶ月前まで完全手書き）
  - 初めての有償案件（過去は無償サポート）

### ✅ 良かった点（そのまま維持）

1. **「このドキュメントについて」セクション**
   - CLIENT_INFOとCLIENT_PROFILEの違いが明確で、迷わない
   - 冒頭に配置されているので、ドキュメントの目的がすぐ分かる

2. **フォルダ構造ガイド**
   - 00_COREとプロジェクトフォルダの役割が明確
   - 実際の構造を反映できる柔軟性がある
   - 「ここに格納するもの/しないもの」が分かりやすい

3. **プロジェクト一覧**
   - 進行中/完了/アーカイブの分類が明確
   - 各プロジェクトに事業主体を明記する欄がある

4. **提案パターンと反応（蓄積型）セクション**
   - 学習型のドキュメントとして非常に有効
   - 効果的だったアプローチ/避けるべきアプローチの整理が実用的

5. **関連リンク・参照ドキュメント**
   - 実際のファイルパスを記載できるので、すぐにアクセスできる

### ⚠️ 改善が必要な点

1. **問題**: 「よく使う単価」セクションが工事関連に特化している
   - **理由**: 黒澤工務店は建設業なので工事単価が必要だが、DOAのようなサービス業には適用しにくい
   - **改善案**: 
     - セクション名を「よく使う単価・料金」に変更
     - 工事単価の例を削除し、より汎用的な形式に
     - または、業種に応じてカスタマイズ可能な例を複数提示

2. **問題**: プロジェクトフォルダが未作成の場合の記入方法が不明確
   - **理由**: DOAの場合、2025/フォルダは空で、プロジェクトフォルダは未作成だった
   - **改善案**: 
     - 「プロジェクトフォルダが未作成の場合」の記入例を追加
     - 「フォルダ: `[プロジェクト名]/`（予定）」のような形式を明示

3. **問題**: 支払条件が未確定の場合の記入方法が不明確
   - **理由**: DOAの場合、初めての有償案件で支払条件が未確定だった
   - **改善案**: 
     - 「未確定」や「案件ごと」という記入例を追加
     - または、該当しない場合は「-」と記入することを明示

4. **問題**: 「担当者」セクションが1名のみ想定されている
   - **理由**: DOAの場合、代表者（雅大氏）と内勤（たけるさん）がいる
   - **改善案**: 
     - 複数の担当者を記載できる形式に変更
     - または、各担当者の役割を明確に記載できる形式に

5. **問題**: 「会社情報」セクションに「Webサイト」があるが、すべてのクライアントにあるわけではない
   - **理由**: 個人事業主や小規模事業者にはWebサイトがない場合がある
   - **改善案**: 
     - 「Webサイト」をオプション項目として明示
     - または、該当しない場合は「-」と記入することを明示

### ➕ 追加すべきセクション

1. **セクション名**: 「関係性の特徴」
   - **理由**: DOAの場合、17年来の元同僚関係という特殊な関係性が重要だが、テンプレートには記載する場所がない
   - **内容案**: 
     - 関係開始時期
     - 関係性の特徴（友人、ビジネスパートナー、元同僚等）
     - 特別な注意事項

2. **セクション名**: 「コミュニケーションスタイル」
   - **理由**: DOAの場合、LINE優先・短文・絵文字多用という特徴があるが、テンプレートには記載する場所がない
   - **内容案**: 
     - 優先連絡手段（LINE、メール、電話等）
     - コミュニケーションの特徴（短文、長文NG、絵文字多用等）
     - 好ましいコミュニケーション方法

3. **セクション名**: 「クイックリファレンス」（よく使うパス・情報を冒頭に）
   - **理由**: 日常業務でよく使う情報をすぐにアクセスできるようにする
   - **内容案**: 
     - よく使うパス（会議記録、契約書、見積書等）
     - よく使う単価・料金
     - 次回アクション

### ➖ 削除・簡略化すべきセクション

1. **セクション名**: 「よく使う単価」の工事関連の例
   - **理由**: 工事関連の例が具体的すぎて、サービス業には適用しにくい
   - **提案**: 削除し、より汎用的な形式に変更

2. **セクション名**: 「完了プロジェクト」の年度別分類
   - **理由**: プロジェクト数が少ない場合、年度別に分ける必要がない
   - **提案**: 簡略化し、必要に応じて年度別に分類できる形式に
