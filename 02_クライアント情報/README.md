# 02_クライアント情報

## 概要
株式会社SKコーム（代表：末武修平）と末武修平個人のクライアント情報管理リポジトリです。
**シンプルで明確な構造**を重視し、AIツール（Claude Code/Cursor）での作業効率を最適化しています。

## ⚠️ 重要：このリポジトリのドキュメント方針

**このリポジトリではクライアントフォルダにREADME.mdを使用しません。**

### ドキュメント構造
```
[クライアント名]/
├── CLIENT_INFO.md          # 実務情報（フォルダ構造、単価、FAQ）
├── 00_CORE/
│   ├── CLIENT_PROFILE.md   # 戦略分析（深層ニーズ、意思決定パターン）
│   ├── source/             # 原データ（LINE、録音等）
│   └── analysis/           # 分析レポート
└── [プロジェクト名]/
    ├── 00_INPUT/           # フラット運用（必要に応じてフォルダ作成）
    ├── 01_OUTPUT/
    │   ├── クライアント向け/
    │   └── 内部用/
    └── 02_ARCHIVE/
```

### なぜREADME.mdを使わないのか？

**理由**：
1. **役割の明確な分離**：CLIENT_INFO（実務）とCLIENT_PROFILE（戦略）の区別が一目瞭然
2. **ファイル名での自己説明性**：`CLIENT_INFO.md`という名前自体が役割を示す
3. **混乱の回避**：複数のREADMEが乱立すると「どれを読むべきか」が不明瞭に

**例外**：
- このルートREADME.md（リポジトリ全体のナビゲーション）のみ使用

## ディレクトリ構造

```
02_クライアント情報/
├── README.md               # このファイル（リポジトリ全体の説明）
├── _inbox/                 # 新規クライアント候補・クライアント不明データ
├── _TEMPLATE/              # テンプレート
│   ├── CLIENT_INFO_TEMPLATE.md
│   └── PROJECT_TEMPLATE/
├── _共通資料/              # クライアント横断の営業資料
├── _自社情報/              # SKコーム・末武修平の会社情報
├── _ARCHIVE/               # 完了・休眠クライアント
├── 黒澤工務店/             # 各クライアントフォルダ（フラット配置）
├── DOA/
├── 翼ビジョン（堀内さん）/
└── [その他クライアント]/
```

### _inbox/の役割
- **新規クライアント候補**: 問い合わせがあったが、まだクライアントフォルダを作成していない段階
- **クライアント不明**: どのクライアントに関連するか不明確なデータ
- **詳細**: [_inbox/README.md](_inbox/README.md)

### フラット配置の理由
- **検索性**：クライアント名で直接アクセス可能
- **シンプル**：カテゴリ分けによる階層を排除
- **柔軟性**：クライアントの関係性が変化しても構造変更不要

## 標準構造の詳細

### CLIENT_INFO.md（各クライアントフォルダ直下）
**役割**：日常業務で使う実務情報

**内容**：
- 基本情報（会社名、担当者、支払条件）
- フォルダ構造ガイド
- ファイル命名規則
- よく使う単価・料金
- プロジェクト一覧
- FAQ・次回提案TODO

### 00_CORE/CLIENT_PROFILE.md
**役割**：戦略立案用の深層分析

**内容**：
- 経営課題の深層分析
- 意思決定パターンの本質
- キーパーソンの心理分析
- 深層ニーズの理解
- 提案戦略

### 00_INPUT/のフラット運用
**方針**：基本はフラット、必要に応じてフォルダ作成

**命名規則**：
```
YYYY-MM-DD_[種別]_[内容].[拡張子]

例：
2025-10-26_会議_AI活用戦略セッション.md
2025-09-15_ヒアリング_社長.md
クライアント提供_ロゴデータ.ai
参考資料_競合分析レポート.pdf
```

**YAMLメタデータ（推奨）**：
Markdownファイルの冒頭に記載することで、AIが内容を理解しやすくなります：
```yaml
---
type: email / meeting / document / resource
from: [送信者/提供元]
date: YYYY-MM-DD
project: [プロジェクト名]
topics: [トピック1, トピック2]
action_required: yes/no
status: pending / reviewed / completed
---
```

**テンプレート**：
- `/_inbox/_TEMPLATE/` に各種テンプレートあり
- EMAIL_TEMPLATE.md, MEETING_TEMPLATE.md 等

**フォルダ作成のタイミング**：
- 同じ種別のファイルが**10件を超えたら**検討
- 例：会議記録10件超 → `会議記録/`フォルダ作成

## 設計思想

### 1. シンプルさの追求
- 複雑な階層構造を排除
- 必要最小限のフォルダ
- 明確な命名規則

### 2. AIツール最適化
- Claude Code/Cursorが理解しやすい構造
- 役割が明確なドキュメント名
- YAMLメタデータでプロジェクト横断検索

### 3. 必要に応じた柔軟性
- 画一的なテンプレートを強制しない
- 10件を超えたらフォルダ作成を検討
- クライアントの特性に応じた調整可能

## 運用ルール

### 新規クライアントの追加
1. トップレベルに`[クライアント名]/`フォルダを作成
2. `_TEMPLATE/CLIENT_INFO_TEMPLATE.md`をコピーして`CLIENT_INFO.md`を作成
3. `00_CORE/`フォルダを作成（CLIENT_PROFILE.md + source/ + analysis/）
4. プロジェクト開始時に`_TEMPLATE/PROJECT_TEMPLATE/`を使用

### クライアントのアーカイブ
以下の場合、`_ARCHIVE/`へ移動：
- プロジェクト完了後、6ヶ月以上連絡なし
- ビジネス関係終了
- 再活性化の可能性は考慮して保管

### データ管理の原則
1. **必要最小限の情報**：情報過多にならないよう注意
2. **検索性の重視**：ファイル名は分かりやすく、日付を含める
3. **定期的な整理**：四半期ごとにアーカイブを見直し
4. **学びの記録**：成功事例、失敗事例から学びを抽出
5. **README.md禁止**：クライアントフォルダではCLIENT_INFO.mdを使用

## Git管理
このリポジトリはGitで管理されています。

### 管理対象
- ✅ テキスト文書（.md, .txt）
- ✅ PDF（契約書、提案書等）
- ✅ 小容量画像（ロゴ、図表）
- ✅ 構造ファイル（YAMLメタデータ含む）

### 除外対象（.gitignore）
- ❌ メディアファイル（動画、音声、大容量画像）
- ❌ 一時ファイル（.DS_Store等）

### コミットメッセージ
- 変更内容を明確に記載
- プロジェクト名を含める
- Co-Authored-By: Claude <noreply@anthropic.com>を活用
