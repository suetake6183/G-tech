# AIセミナー設計についての共有

**作成日**: 2025年12月6日
**共有先**: 黒澤さん
**目的**: 12月5日打ち合わせを踏まえたセミナー設計の思考プロセスを共有し、意見をいただく

---

## 1. 12月5日の打ち合わせで見えたこと

### 当初の想定と実態のズレ

当初は「AIをほとんど触ったことがない人向けの入門セミナー」を想定していました。

しかし、打ち合わせで参加者の実態を聞いてみると、状況が異なっていました。

| 当初の想定 | 実際の参加者の状態 |
|-----------|------------------|
| AIに触れたことがない | Plaud.ai等で既に議事録を取っている |
| 基本操作から教える必要がある | 調べ物・文書作成には使っている |
| 「AIとは何か」から始める | 「使っているが、これでいいのか」と不安 |

### 参加者から出てきた発言

**石川さん**: Plaud.aiを営業2名にも配布済み。「AIなしではダメ」の領域を目指したい。

**松田さん**: 社員向けに導入したい。AIに乗り遅れる焦りがある。

**宮森さん**: Plaud.aiで議事録作成している。経営情報と社員情報の切り分けに関心。

### 発言の中にあった矛盾

打ち合わせの発言を分析すると、いくつかの矛盾が見えました。

| 表面的な発言 | 実態 |
|------------|------|
| 「社員に使わせたい」 | 自分が価値を実感できていない |
| 「焦りがある」 | 具体的な行動に移せていない |
| 「使っている」 | 「これでいいのか」と不安なまま |

この矛盾の根本原因は、**「AIを使わなきゃいけない」という外部からのプレッシャーと、「自分にとって何が嬉しいのか」という内発的動機がつながっていない**ことだと考えました。

## 2. セミナー設計の方針転換

### 当初のメインテーマ「自社分析・言語化」について

当初は「自社を言語化することがAI活用の土台になる」というテーマを考えていました。

しかし、参加者の実態を見ると、このテーマは時期尚早だと判断しました。

**理由**:
- 参加者は既にAIを触っている
- 抽象的な「言語化の重要性」より、具体的な「これができる」を見たい
- まず「AIでここまでできるんだ」という実感が先にないと、言語化の必要性が腹落ちしない

### 新しい方向性

「AIでここまでできる」を具体的なデモで見せ、参加者が価値を実感できる状態を作る。

---

## 3. コンテンツの選定基準

どのコンテンツをセミナーに入れるかを決めるにあたり、4つの基準を設けました。

| 基準 | 理由 |
|------|------|
| 1. 既にやっていることの延長 | 新しいことより「今やっていることがもっと良くなる」方が価値を感じやすい |
| 2. 経営者自身が役立つと感じる | 対象が経営者なので、経営者目線で刺さるものを選ぶ |
| 3. デモで差が見せやすい | ビフォーアフターが明確だと「違い」が伝わる |
| 4. 社員にシェアしたくなる | 打ち合わせで「社員に使わせたい」という声が多かった |

4番目の基準は、松田さんの「社員向けに導入したい」という発言から追加しました。経営者が価値を感じて終わりではなく、社内に持ち帰って展開できるものが求められていると判断しました。

---

## 4. 選定したコンテンツ（3つ）

### コンテンツ1: 議事録の精度向上

**選んだ理由**:
- 参加者の多くがPlaud.ai等で既に議事録を取っている
- 既にやっているからこそ、「もう一段上」を見せると差が明確
- 社員にもそのまま展開できる

**デモで見せること**:
- 単なる文字起こしと、構造化された議事録の違い
- 発言者別の整理、決定事項、宿題の切り分け

**素材**:
- 事前に用意したサンプル音声、または参加者の議事録（許可を取得）

---

### コンテンツ2: 営業フィードバック

**選んだ理由**:
- 石川さんが「スコープ×ChatGPTで営業会議」をやっていると発言
- 部下に直接言いにくいことをAIが代わりに指摘する、という切り口は経営者に刺さる
- 営業担当者にも直接役立つため、社員への展開がしやすい

**デモで見せること**:
- 営業録音をAIに分析させる
- 「話しすぎ」「顧客の反応を拾えていない」等の客観的フィードバック

**素材**:
- ロールプレイで作成
- 実際の営業録音を提供いただける企業があれば使用

---

### コンテンツ3: 経営の壁打ち

**選んだ理由**:
- 経営者は孤独。部下には聞けないことがある
- コンサルに頼むほどでもない日常的な悩みをぶつける相手がいない
- AIなら忖度なく批判的な意見をもらえる

**進め方**:
1. 末武が自分の経営課題（過去に悩んでいたが解決済みのもの）を使ってデモ
2. 参加者が各自のスマホで自分の課題を入力して壁打ち体験
3. 感想を共有してくれる人がいれば聞く

**この形式にした理由**:
- 典型的な課題でデモしても「他人事」になる
- 自分の課題で体験しないと「自分ごと」にならない
- ただし、いきなり課題を出してもらうのはハードルが高いので、末武が先に見本を見せる

---

## 5. 優先度を下げたコンテンツ

| コンテンツ | 下げた理由 |
|-----------|-----------|
| 図面の整合性チェック | 宮森さんが興味を示していたが、デモの準備が重い。個別フォローで対応する方が効率的 |
| 自社の言語化→集客・採用 | 抽象度が高く、初回セミナーで入れると散漫になる |
| 情報セキュリティの切り分け | Q&Aや個別相談で対応する方が適切 |

## 6. セミナー構成案

| 時間 | 内容 |
|------|------|
| 10分 | 導入: 参加者の現状確認 |
| 20分 | デモ1: 議事録の精度向上 |
| 20分 | デモ2: 営業フィードバック |
| 25分 | デモ＋ワーク: 経営の壁打ち |
| 10分 | まとめ: 明日からできること |
| 15分 | 質疑応答 |

**合計**: 約100分
