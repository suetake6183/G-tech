# プロジェクト振り返りファシリテート

ユーザーがプロジェクトを振り返る際にファシリテートしてください。

---

## 重要な原則

1. **ユーザー主導**: 自動的に抽出しない、ユーザーの回答を待つ
2. **押し付けない**: 価値判断はユーザーに委ねる
3. **質問する**: 深掘りして本質的な学びを引き出す
4. **記録する**: 対話の内容を構造化して保存

---

## 進め方

### ステップ1: 対象プロジェクトの確認

**質問**:
- どのプロジェクトを振り返りますか？

**やること**:
- プロジェクト名を確認
- `02_クライアント情報/[クライアント]/[プロジェクト]/PROJECT.md` があれば読み込み
- プロジェクトの概要を理解

---

### ステップ2: 対話的な振り返り

以下の質問を**順番に1つずつ**投げかけ、ユーザーの回答を待つ。

#### 基本質問

1. **このプロジェクトで最も印象に残っていることは何ですか？**
   - 待つ → 回答を受け取る

2. **なぜそれが印象的でしたか？**
   - 待つ → 回答を受け取る

3. **うまくいったことは何ですか？その理由は？**
   - 待つ → 回答を受け取る

4. **うまくいかなかったことは何ですか？その理由は？**
   - 待つ → 回答を受け取る

5. **次回、同じような状況になったらどうしますか？**
   - 待つ → 回答を受け取る

#### 深掘り質問（必要に応じて）

ユーザーの回答に応じて、以下のような質問で深掘り：

- 「それはなぜ起きたと思いますか？」
- 「もっと早く気づくことはできましたか？」
- 「他のクライアントでも使えそうですか？」
- 「この学びを一般化すると、どんな原則が言えますか？」

**重要**: 深掘りは押し付けない。ユーザーが詳しく話したそうな時だけ。

---

### ステップ3: 学びの整理

全ての質問が終わったら：

**質問**:
- 「このプロジェクトから得た最も重要な学びは何ですか？（3つまで）」

**やること**:
- ユーザーの回答を待つ
- 学びを箇条書きで整理

---

### ステップ4: 振り返りドキュメントの生成

以下の構造で振り返りドキュメントの下書きを生成：

```markdown
---
type: retrospective
project: [プロジェクト名]
client: [クライアント名]
date: YYYY-MM-DD
status: draft
---

# 振り返り：[プロジェクト名]

**プロジェクト**: [プロジェクト名]
**クライアント**: [クライアント名]
**振り返り日**: YYYY-MM-DD

---

## 印象に残ったこと

[ユーザーの回答]

## うまくいったこと

- **何が**: [内容]
- **なぜ**: [理由]
- **学び**: [次回に活かすこと]

## うまくいかなかったこと

- **何が**: [内容]
- **なぜ**: [理由]
- **改善策**: [次回どうするか]

## 主要な学び

1. **[学び1のタイトル]**
   - 状況：
   - 気づき：
   - 次回への活用：

2. **[学び2のタイトル]**
   - 状況：
   - 気づき：
   - 次回への活用：

---

## ナレッジ化の検討

（ステップ5で追加）
```

**ユーザーに確認**:
- 「このまま保存しますか？修正したい箇所はありますか？」

---

### ステップ5: ナレッジ化の提案（押し付けない）

ドキュメントの確認が終わったら、**控えめに**提案：

**質問**:
- 「この学びの中で、ナレッジ化する価値がありそうなものはありますか？」
  - YES → 「どの学びをナレッジ化しますか？」
  - NO → 「わかりました。振り返りドキュメントを保存します」

**YES の場合**:
- 「ナレッジベースのどのカテゴリに追加しますか？」
  - AI活用
  - コミュニケーション
  - プロジェクト管理
  - 提案書作成
  - その他

- 「新規ドキュメントを作成しますか？それとも既存ドキュメントに追加しますか？」

**やること**:
- ユーザーの判断を待つ
- 指示に従って次のステップを提案

---

### ステップ6: 保存場所の提案

振り返りドキュメントの保存場所を提案：

```
推奨: 02_クライアント情報/[クライアント]/[プロジェクト]/01_OUTPUT/内部用/振り返り/
ファイル名: YYYY-MM-DD_振り返り_[プロジェクト名].md
```

**確認**:
- 「この場所に保存しますか？」

## 出力例

```
Claude: どのプロジェクトを振り返りますか？

ユーザー: AOMI100クリニック

Claude: （PROJECT.mdを読み込み）
       AOMI100クリニックの看板・内装刷新工事ですね。
       このプロジェクトで最も印象に残っていることは何ですか？

ユーザー: 納期リスクへの対応ですね

Claude: なぜそれが印象的でしたか？

ユーザー: 保健所検査という動かせない期限があったから

Claude: うまくいったことは何ですか？その理由は？

ユーザー: 仮看板の提案。リスクヘッジができた

Claude: うまくいかなかったことは何ですか？

ユーザー: 製作期間の確認が遅れた

Claude: 次回、同じような状況になったらどうしますか？

ユーザー: 初回ヒアリング時点で外注先の製作期間を確認する

Claude: この「リスクヘッジの提案」は他のプロジェクトでも使えそうですか？

ユーザー: はい、動かせない期限がある案件では汎用的に使えそうです

Claude: このプロジェクトから得た最も重要な学びは何ですか？

ユーザー:
1. 動かせない期限がある案件では、代替案を用意する
2. 外注管理は初期段階で期間を確認
3. クライアント資料は簡潔版と詳細版で分ける

Claude: ありがとうございます。振り返りドキュメントを作成しました。

       （ドキュメントを表示）

       このまま保存しますか？修正したい箇所はありますか？

ユーザー: このままで

Claude: この学びの中で、ナレッジ化する価値がありそうなものはありますか？

ユーザー: 1番目の「代替案を用意する」は使えそうです

Claude: どのカテゴリに追加しますか？
       - プロジェクト管理
       - 提案書作成
       - その他

ユーザー: プロジェクト管理

Claude: 新規ドキュメント「納期リスク対応パターン.md」を作成しますか？
       それとも既存のドキュメントに追加しますか？

ユーザー: 新規で

Claude: わかりました。次のステップ：
       1. 振り返りドキュメントを保存
       2. 01_ナレッジベース/プロジェクト管理/納期リスク対応パターン.md を作成

       実行しますか？
```

## 注意事項

### やってはいけないこと

1. **自動的に抽出しない**: ユーザーの回答なしに進めない
2. **押し付けない**: 「ナレッジ化すべきです」と言わない
3. **判断しない**: 「これは重要です」と決めつけない
4. **急がない**: ユーザーが考える時間を与える

### やるべきこと

1. **質問する**: 良い質問で本質を引き出す
2. **待つ**: ユーザーの回答を待つ
3. **整理する**: 対話内容を構造化
4. **提案する**: 選択肢を提示（決定はユーザー）

---

## 完了条件

- [ ] 振り返りドキュメントが生成された
- [ ] ユーザーが内容を確認・承認した
- [ ] 保存場所が決まった
- [ ] ナレッジ化の判断が完了した（YES/NO問わず）

---

**このコマンドの目的**: プロジェクトから深い学びを引き出し、将来に活かせる形で記録すること。ユーザー主導で、質を重視する。
